## 1. 產品概述

### 1.1 什麼是 AI 管家？

AI 管家是發票 App 中的智慧消費分析功能。透過分析用戶的電子發票數據（通路、金額、頻率、時間），以對話式 chatbot 的形式，主動挖掘用戶自己都沒意識到的消費洞察，並提供具體的省錢建議。

### 1.2 核心價值

| 面向 | 描述 |
|------|------|
| **對用戶** | 「原來我的消費長這樣」——發現隱藏的消費規律、爬升的均價、正在形成的新習慣 |
| **對產品** | 提升發票 App 的黏性和打開頻率，從「工具」進化為「顧問」 |
| **差異化** | 不是告訴你花了多少（記帳 App 做得到），而是告訴你「為什麼花這麼多」和「怎麼花得更聰明」 |

### 1.3 設計原則

| 原則 | 說明 | 反例 |
|------|------|------|
| **有痛有感** | 每個洞察都要讓用戶感到驚訝或「原來如此」 | 不能只是列出消費數字 |
| **針對性** | 根據每個人的真實數據動態生成，不是通用模板 | 不能所有人看到一樣的問題 |
| **不評判** | 描述行為而非下結論，讓用戶自己決定 | 不說「你外食太多」，說「每 2.9 天叫一次外送」 |
| **體驗型比較** | 省錢換算用旅行、聚餐、健身等體驗，不用 3C 商品 | 不說「= 一台 iPhone」 |
| **正負平衡** | 不只指出問題，也肯定做得好的地方 | 不能全是「你花太多」 |

## 2. 用戶流程

### 2.1 進入方式

```
App Tab 列 → 點擊「AI 管家」
  → 如果未登入 → 登入頁（輸入手機號碼）
  → 登入成功 → 載入 RTDB 用戶數據
  → 進入 AI 管家 Chatbot
```

### 2.2 登入機制

- 輸入手機號碼（09xx-xxx-xxx）
- 後端查詢 Firebase RTDB `/users/{phone}`
- 成功：進入主頁面，資料傳入各頁面元件
- 失敗：顯示「找不到此手機號碼的會員資料」

### 2.3 Chatbot 操作流程（樹狀敘事鏈）

```
開場（3 句漸進出現）
  → 第 3 句連結 Hook 1 的主題
  → 只顯示 Hook 1 按鈕

用戶點擊 Hook 1
  → 顯示 Insight 卡片（大數字 + 排行 + 💡觀察）
  → 出現 2 個追問按鈕

用戶點擊追問 A
  → 打字動畫回答
  → 出現 2 個深追問按鈕（紫色）
  → 追問 B 仍然可點（標示「或回到其他追問」）

用戶點擊深追問 A1
  → 打字動畫回答
  → 深追問 A2 仍然可點
  → AI 發送橘色引導句 → Hook 2 解鎖

... 重複直到 Hook 4

Hook 4 深追問完 → AI 發送總結
```

### 2.4 層級結構

```
Hook（Insight 卡片 + 數據視覺化）           ← Level 1
  ├─ 追問 A（打字動畫回答）                  ← Level 2
  │   ├─ 深追問 A1（打字動畫回答）            ← Level 3
  │   └─ 深追問 A2（打字動畫回答）
  ├─ 追問 B（打字動畫回答）
  │   ├─ 深追問 B1（打字動畫回答）
  │   └─ 深追問 B2（打字動畫回答）
  └─ [任一深追問完成] → 引導句 → 下一個 Hook 解鎖
```

### 2.5 自由輸入

除了預設問題，用戶可以在底部輸入框打任何問題。自由輸入透過 Cloudflare Workers AI（Llama 3.1 8B）回答，Context 包含用戶的消費摘要數據。

## 3. Insight Engine — 動態洞察生成

### 3.1 架構

每個用戶登入後，前端即時計算所有可能的洞察候選，按分數排序，挑出前 4 個作為 Hook。不同用戶看到不同的 Hook 組合。

### 3.2 洞察類型（7 種）

| 類型 | ID | 觸發條件 | 分數 | Hook 問題 |
|------|-----|---------|------|----------|
| **消費依賴** | `freq` | 最高頻品牌 ≥5 次 | 80+ | 「你最離不開什麼？」 |
| **消費擴張** | `growth` | 品牌成長 >50% 或從零新增 | 90+ | 「什麼消費正在悄悄擴張？」 |
| **均價爬升** | `creep` | 品牌均價上升 >15%（排除帳單） | 80+ | 「哪裡每次花的錢越來越多？」 |
| **隱形高消費** | `hidden` | 中低頻但佔總消費 >3%（排除帳單） | 75+ | 「哪裡在不知不覺吃掉預算？」 |
| **類別集中** | `dominance` | 單一類別 >15%（排除「其他」和帳單） | 65+ | 「你的錢 X% 花在哪？」 |
| **未來預估** | `projection` | 永遠產生 | 60+ | 「照這樣下去，一年會花多少？」 |
| **正向肯定** | `positive` | 有品牌均價穩定或下降 | 72+ | 「我有哪裡做得不錯？」 |
| **省錢方案** | `save` | 有 ≥2 個省錢策略可計算 | 85 | 「我可以怎麼聰明省錢？」 |

### 3.3 排除邏輯

- **帳單類別排除**：電費、水費、瓦斯費、電信費、網路 → 不參與「隱形高消費」「均價爬升」等分析（這些是固定支出，不是行為問題）
- **平台費標記**：Uber Eats / foodpanda 等外送平台，若均價 < $50，標註為「發票為平台服務費」（實際餐費另計）
- **「其他」類別**：通路對照表中未分類的通路，不參與類別相關分析

### 3.4 分數計算邏輯

```
每種洞察的基礎分 + 加權：
- GROWTH: 90（新增）或 70（成長）+ min(近期次數, 30)
- CREEP: 80 + min(均價漲幅, 20)
- FREQUENCY: 80 + min(最高頻次數, 20)
- SAVE: 85（固定，幾乎一定入選）
- POSITIVE: 72 + 好表現品牌數 × 3
- DOMINANCE: 65 + 佔比百分點
- PROJECTION: 60 + (趨勢上升 ? 20 : 10)
```

### 3.5 開場連動

開場第 3 句根據 Hook 1 的類型動態生成，確保敘事連貫：

| Hook 1 類型 | 開場第 3 句範例 |
|------------|---------------|
| growth | 「全家的消費正在快速增加——前期 4 次到近期 18 次」 |
| freq | 「你跟 Uber Eats 的關係很穩定——每 3.8 天一次」 |
| creep | 「你去全聯的頻率沒變，但每次花的錢從 $500 升到 $700」 |
| positive | 「你在全聯的消費越來越精準——均價從 $700 降到 $500」 |
| save | 「我發現有 $XX,XXX/年可以不改變生活就省下來」 |

### 3.6 引導句（Bridge Sentences）

Hook 之間的引導句根據前後兩個 insight type 的組合動態生成。範例：

| 前 → 後 | 引導句 |
|---------|--------|
| growth → creep | 「看到了什麼在擴張。同時有些通路的均價也在悄悄爬升——」 |
| freq → save | 「知道了你的消費依賴。那有什麼辦法可以花得更聰明？」 |
| creep → positive | 「均價在爬升的要注意。但也有些通路你做得很好——」 |

## 4. 省錢比較框架（消費心理學）

### 4.1 三層框架

基於行為經濟學研究（機會成本忽視、損失規避、體驗型消費），省錢換算分三層：

| 層級 | 心理機制 | 範例 |
|------|---------|------|
| **Layer 1: 自我錨點** | Self-anchoring（用自己的消費做比較最有感） | 「等於你去 7-ELEVEN 70 次」 |
| **Layer 2: 體驗替代** | 體驗消費的幸福感 > 物質消費 | 「一趟東京 3 天快閃」「跟朋友吃 20 次好餐廳」 |
| **Layer 3: 日均損失** | Loss Aversion（失去的痛是得到的 2 倍） | 「每天 $28 正在溜走——等於每天丟掉一杯路易莎拿鐵」 |

### 4.2 參考價格（2026 年）

| 品項 | 價格 | 來源 |
|------|------|------|
| 路易莎美式（中杯） | $55 | 路易莎官方菜單 |
| 路易莎拿鐵（中杯） | $100 | 路易莎官方菜單 |
| 星巴克拿鐵（大杯） | $155 | 星巴克官方菜單 |
| 台北好餐廳（人均） | $600 | 台北餐廳調查 |
| 健身房月費（中階） | $988 | World Gym |
| 國內兩天一夜 | $5,000/人 | 國旅平均花費 |
| 東京來回廉航 | $5,000 | KAYAK/Skyscanner |
| 東京來回傳統航空 | $14,000 | 星宇航空 |

### 4.3 語氣原則

| 場景 | ❌ 不要 | ✅ 要 |
|------|--------|------|
| 描述高頻消費 | 「你的外送費高達 $6,930」 | 「你跟 UberEats 的關係很穩定——每 2.9 天一次」 |
| 描述趨勢上升 | 「你的咖啡花費暴增 187%」 | 「路易莎正在悄悄變成你的日常必需品」 |
| 描述佔比 | 「你的外食佔比過高，達 38%」 | 「你花在吃喝上的比例，從 27% 升到了 41%」 |
| 給建議 | 「建議你減少外食」 | 「如果每週少叫 2 次，一年差 $31,680。你自己決定」 |
| 肯定用戶 | （不提） | 「你在全聯的均價 6 個月沒漲——你是有紀律的採購者」 |

## 5. 省錢方案（SAVE_PLAN）

### 5.1 策略清單

AI 管家會根據用戶數據，自動計算並排序以下省錢策略：

| 策略 | 觸發條件 | 省錢邏輯 | 難度 |
|------|---------|---------|------|
| 🏪→🛒 超商改超市 | 超商 ≥10 次 + 有超市消費 | 同品項超市便宜 20-30% | 低 |
| 📉 控制均價爬升 | 有均價上升 >15% 的品牌 | 回到之前的均價水準 | 低 |
| 📅 減少最高頻通路 | 最高頻 ≥10 次 + 每週 ≥2 次 | 每週少去 1 次 | 中 |
| 🔄 合併同類別通路 | 同類別有 ≥2 個品牌 | 集中一家累積回饋 | 低 |
| 🆕 控制新習慣 | 成長 >200% 或從零新增 | 頻率減半 | 中 |

### 5.2 追問結構

```
「我可以怎麼聰明省錢？」
  ├─ 「最無痛的第一步是什麼？」
  │   ├─ 「做了之後下一步呢？」
  │   └─ 「會不會影響生活品質？」
  └─ 「全部做到的話呢？」
      ├─ 「哪個 CP 值最高？」
      └─ 「我的消費整體算健康嗎？」
```

## 8. AI 模型注意事項（給工程師）

### 8.1 如果要換成 LLM 生成回答

目前 Hook/追問/深追問的回答都是前端 JS 預計算的（insightEngine.js），不經過 AI 模型。如果未來要改成 LLM 生成：

#### System Prompt 要求

```
你是發票 AI 管家，根據用戶的真實消費數據回答問題。

規則：
1. 只能引用 Context 中提供的數據，不能編造
2. 用繁體中文，語氣親切但專業
3. 描述行為而非評判（「每 2.9 天叫一次」而非「外送太多」）
4. 省錢建議要具體：通路名 + 金額 + 頻率
5. 比較用體驗（旅行、聚餐）而非物質（iPhone）
6. 正負平衡：指出問題也要肯定做得好的
7. 控制在 200 字以內
8. 適度用 emoji
```

#### Context 需包含

```
- 發票總數和總金額
- 前 5 大通路（品牌名、次數、金額、類別）
- 前 5 大類別（名稱、金額、佔比）
- 月趨勢數據
- 前後期變化（哪些在增加、哪些在減少）
```

#### 注意事項

- Uber Eats / foodpanda 的發票金額是平台服務費，不是餐費。LLM 不要說「你在 Uber Eats 花了 $1,176」而要說「你叫了 55 次外送（發票為平台費 $1,176）」
- 電費、電信費等固定支出不要當作「可以省的」
- 超市量販單價高是正常的（大量採購），不要標記為問題。只有均價在「上升」才是問題
- 不要給出超過用戶數據範圍的建議（如：推薦信用卡、推薦投資）

### 8.2 前端預計算 vs LLM 的取捨

| 面向 | 前端預計算（目前） | LLM 生成 |
|------|-------------------|---------|
| 速度 | 即時 | 2-5 秒延遲 |
| 準確度 | 100%（純數學） | 可能有幻覺 |
| 靈活度 | 只能回答預設問題 | 任何問題都能回答 |
| 成本 | $0 | 每次 API 呼叫 $0.001-0.01 |
| 建議 | Hook/追問/深追問用預計算 | 自由輸入用 LLM |

### 9.5 操作規則

1. 開場後只顯示 Hook 1（不一次全給）
2. 點追問後，該追問消失，兄弟追問保留
3. 深追問回答後，兄弟深追問保留，上層未點追問也保留
4. 任一深追問完成後，觸發引導句 + 解鎖下一個 Hook
5. Hook 按鈕只顯示已解鎖且未使用的
6. 自由輸入任何時候都可以用

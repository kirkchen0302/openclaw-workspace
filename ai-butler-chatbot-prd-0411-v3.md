## 1. 版本差異摘要（vs 0409_v3）

| 面向 | 0409_v3 | 0411_v3 |
|------|---------|---------|
| 資料來源 | RTDB `invoices`（通路+金額） | RTDB `invoices_v2`（通路+金額+**品項明細**） |
| 品項分類 | 無 | `itemClassifier.js`（30+ 品類） |
| Hook 生成 | 固定 4 主題 | **Insight Engine 動態選擇**，每人不同 |
| Hook 數量 | 4 個 | 3-5 個（依資料豐富度） |
| Hook 解鎖 | 順序解鎖（敘事鏈） | **全部一開始就可點** |
| 省錢比較 | 通用 3C 商品 | **消費心理學 3 層框架**（自我錨點/體驗替代/日均損失） |
| 外送平台 | 參與所有分析 | **排除**（發票為服務費，非實際消費） |
| 新用戶 | 無處理 | **Onboarding 引導**（載具/歸戶/使用/對獎/小工具） |

## 2. 資料架構

### 2.1 RTDB 結構（v2）

```
/users/{phone}/
├── invoices_v2[]              ← 新增：含品項明細
│   ├── issued_at              // "2026-04-07T23:03:00"
│   ├── yearMonth              // "2026-04"
│   ├── day                    // "7"
│   ├── week                   // "週二"
│   ├── hour                   // 23
│   ├── shop                   // "麥當勞"（BQ channel 表的中文名）
│   ├── amount                 // 163
│   └── items[]                // ← 品項明細
│       ├── name               // "餐-六塊雞M"
│       ├── price              // 68
│       └── qty                // 1
├── invoiceCount_v2            // v2 發票數
├── totalAmount_v2             // v2 總金額
├── itemCount_v2               // 品項總數
├── invoices[]                 // v1（保留，向下相容）
└── ... （其他既有欄位）
```

### 2.2 品項分類器（itemClassifier.js）

關鍵字匹配，30+ 品類：

| 大類 | 品類 | 關鍵字範例 |
|------|------|----------|
| 飲品 | 咖啡、茶飲、手搖飲、瓶裝飲料、乳製品 | 拿鐵、奶茶、氣泡水、豆漿、LP33 |
| 食物 | 速食餐點、便當/飯糰、麵包/烘焙、零食/餅乾、滷味/小食、生鮮蔬果、生鮮肉品 | 漢堡、飯糰、洋芋片、茶葉蛋 |
| 日用 | 衛生紙/面紙、洗髮/沐浴、口腔清潔、美妝保養、生理用品 | 舒潔、牙膏、面膜 |
| 服務 | 外送服務費、訂閱服務、加油、停車 | 平台費、Uber One、無鉛汽油 |

### 2.3 排除邏輯

| 排除對象 | 原因 | 影響範圍 |
|---------|------|---------|
| **外送平台**（ubereats、foodpanda） | 發票是服務費/平台費，非實際餐費 | 所有品項分析、opener、Hook 排行 |
| **網購/量販**（momo、蝦皮、酷澎、好市多） | 箱購/大包裝，單價不可比 | PRICE_GAP（已移除） |
| **帳單類**（電費、水費、電信費） | 固定支出，非行為問題 | 隱形高消費、均價分析 |

### 2.4 通路名稱處理

v2 的 `shop` 來自 BigQuery channel 表（如「全聯」「麥當勞」），**不需要經過 shopMapping.js 的 resolveShop**。僅用 shopMapping 取得 `category_lv1`（超市、速食等）。

## 3. Insight Engine — 動態洞察生成

### 3.1 架構

```
用戶登入 → RTDB 返回 invoices_v2[]
  → computeStats(): 品牌/品類/月份/品項統計
  → detectInsights(): 產生所有候選 insight → 選出最佳組合
  → 選最震撼的事實當開場 → 對應 Hook 排到第一位
  → 渲染 Chatbot UI
```

### 3.2 Insight 類型（8 種候選）

| 類型 | ID | 觸發條件 | 分數 | Hook 問題範例 |
|------|-----|---------|------|-------------|
| **習慣扣款** | `autopay` | 品項買≥3次且≥3個 | 95+ | 「這 $X 都花在哪些品項？」 |
| **品項/飲料分析** | `items` | 有品項明細 | 82+ | 「這 164 杯飲料都花在哪？」 |
| **省錢方案** | `save` | 有≥2個策略 | 85 | 「怎麼省最有感？」 |
| **訂閱檢視** | `subscription` | ≥2個訂閱 | 78+ | 「我的訂閱還值得嗎？」 |
| 消費擴張 | `growth` | 品牌成長>50% | 90+ | 「什麼消費正在擴張？」 |
| 均價爬升 | `creep` | 均價上升>15% | 80+ | 「哪裡每次花的錢越來越多？」 |
| 消費依賴 | `frequency` | 最高頻≥5次 | 80+ | 「你最離不開什麼？」 |
| 正向肯定 | `positive` | 有均價穩定/下降 | 72+ | 「我有哪裡做得不錯？」 |

### 3.3 Hook 選擇邏輯

**有品項資料時（v2）：**
1. 核心 3 個：`autopay` → `items` → `save`（強制入選）
2. 第 4 個：從剩餘候選中取分數最高的（growth/creep/frequency/positive 等）
3. 第 5 個：`subscription`（僅當≥2個訂閱時出現）

**無品項資料時（v1）：**
按分數排序取前 4 個

### 3.4 開場選擇

開場從所有候選中挑**最震撼的事實**，然後把對應的 Hook 排到第一位：

| 候選 | 分數邏輯 | 範例 |
|------|---------|------|
| 飲料總量 | min(杯數, 85) | 「你喝了 164 杯飲料」 |
| 年省金額 | min(金額/500, 80) | 「每年有 $23,400 可以省」 |
| 習慣品項數 | min(品項數×2, 70) | 「36 個品項你反覆在買」 |
| 訂閱 | 固定 55 | 「3 個訂閱在自動扣款」 |
| 通路頻率 | min(次數, 50) | 「每 3.7 天去一次 7-11」 |

**開場 → Hook 按鈕的對應原則：開場告訴你 WHAT，Hook 帶你看 WHY/WHERE/HOW**

| 開場（震撼事實） | Hook 按鈕（帶你更深） |
|---|---|
| 「你喝了 **164 杯**飲料」 | 「這 **164 杯**都花在哪？」 |
| 「每年有 **$23,400** 可以省」 | 「怎麼省**最有感**？」 |
| 「**36 個品項**反覆在買，累計 $X」 | 「這 **$X** 都花在哪些品項？」 |
| 「**3 個訂閱**在自動扣款 $770」 | 「我的訂閱還**值得**嗎？」 |

## 4. Hook 詳細規格

### 4.1 AUTOPAY —「習慣自動扣款」

**大數字**：品項數量（如「36 個」）
**排行**：反覆購買品項，顯示「次數 + 累計金額 + 通路」（不用月費框架）
**追問**：
- 「哪些想保留、哪些該檢視？」→ 健康/營養 ✅ vs 零食/飲料 🤔
  - 深追問：「減掉順手買的能省多少？」「每家店的固定菜單？」
- 「哪些品項是我沒意識到一直在買的？」→ 跳過前 3 名，找隱藏重複品
  - 深追問：「這些加起來一年多少？」「有更划算的買法？」

### 4.2 ITEMS —「飲料/品項分析」

**飲料 ≥ 20 杯時**：聚焦飲料（咖啡/茶飲/瓶裝/乳製品）
**飲料 < 20 杯時**：退回全品類分析

**飲料模式**：
- 大數字：杯數
- 排行：只顯示飲料品類
- 提示：每杯均價 + 最常喝的品項名
- 追問：「我在 {店} 都買什麼？」「飲料類花了多少？」

### 4.3 SAVE —「省錢方案」

5 種策略按「省最多 × 最不費力」排序：

| 策略 | 觸發條件 | 難度 |
|------|---------|------|
| 超商改超市 | 超商≥10次 + 有超市 | 低 |
| 控制均價爬升 | 有均價上升品牌 | 低 |
| 減少最高頻通路 | 每週≥2次 | 中 |
| 合併同類通路 | 同類≥2個品牌 | 低 |
| 控制新習慣 | 成長>200% | 中 |

### 4.4 SUBSCRIPTION —「訂閱檢視」（第 5 Hook，≥2 訂閱才出現）

**偵測邏輯**：嚴格 keyword match（訂閱/月費/會員訂閱/Uber One/pandapro）
**排除**：一次性購物（Apple 硬體、蝦皮商品）
**重點**：月費 → 年度加總的衝擊感（不做損益兩平分析）
**追問**：
- 「這些訂閱都還值得嗎？」→ 自問「上個月用了幾次？」
- 「放大到一年來看呢？」→ 月費×12 + 個人化體驗換算

## 5. 省錢比較框架（消費心理學）

| 層級 | 心理機制 | 範例 |
|------|---------|------|
| **Layer 1: 自我錨點** | Self-anchoring | 「等於你去 7-ELEVEN 70 次」 |
| **Layer 2: 體驗替代** | 體驗 > 物質 | 「一趟東京 3 天快閃」「跟朋友吃 20 次好餐廳」 |
| **Layer 3: 日均損失** | Loss Aversion | 「每天 $28 正在溜走——等於每天一杯路易莎拿鐵」 |

**2026 參考價格**：路易莎拿鐵 $100、星巴克拿鐵 $155、好餐廳人均 $600、健身房 $988/月、國內兩天一夜 $5,000、東京廉航 $5,000

## 6. 語氣原則

| 場景 | ❌ 不要 | ✅ 要 |
|------|--------|------|
| 描述高頻消費 | 「你的外送費高達…」 | 「你跟 XX 的關係很穩定」 |
| 描述趨勢上升 | 「暴增 187%」 | 「正在悄悄變成你的日常」 |
| 給建議 | 「建議你減少」 | 「每週少 2 次差 $X，你自己決定」 |
| 訂閱 | 「在浪費」 | 「值得重新想想」「還值得嗎？」 |
| 肯定 | （不提） | 「你在 XX 花得越來越精準」 |
| 月費框架 | 「$51/月在爭鮮」（壽司單盤不該算月費） | 「9 次 共 $360 在爭鮮」 |

## 7. 新用戶 Onboarding

### 7.1 觸發條件

`invoiceCount === 0` 或 `invoiceCount < 10` 或 RTDB 無此用戶

### 7.2 引導內容（5 主題）

1. **📱 什麼是載具？** — 概念、好處、安全性
2. **🔗 怎麼歸戶？** — 互動式：選擇付款方式（信用卡/悠遊卡/行動支付/現金）→ 各自的步驟 → 「✅ 我設好了」行動按鈕
3. **🔲 結帳怎麼用？** — 先掃載具再付款、小工具快捷
4. **🏆 怎麼對獎？** — 雙數月 25 號、自動匯入設定
5. **⚡ 桌面小工具** — iPhone / Android 設定步驟

### 7.3 呈現形式

混合式：AI 打招呼 → 主題按鈕 → 步驟卡片 + 對話追問 → 行動確認按鈕

## 8. UI/UX 規格

### 8.1 Hook 按鈕行為

- **所有 Hook 一開始就全部可點**（不順序解鎖）
- 用戶可自由選擇最感興趣的開始探索
- 所有 Hook 都探索完後，顯示 AI 總結
- 每個 Hook 內有 2 追問 × 2 深追問（3 層樹狀）
- 點追問後，兄弟追問保留可點
- 點深追問後，兄弟深追問 + 上層未點追問都保留

### 8.2 按鈕樣式

| 層級 | 樣式 | 說明 |
|------|------|------|
| Hook 按鈕 | 藍色邊框膠囊，水平滾動 | 開場後顯示所有 |
| 追問按鈕 | 深灰邊框全寬 | Hook 展開後 |
| 深追問按鈕 | 紫色邊框全寬 | 追問回答後 |
| 行動按鈕（Onboarding） | 綠色「✅ 我設好了」/ 灰色「⏰ 等等再設定」 | 歸戶引導 |

### 8.3 發票頁（我的發票）

v2 資料新增顯示：
- 消費時間（HH:MM）
- 品項標籤（品名 + 單價，最多顯示 3 個，超過顯示 +N）

## 10. AI 模型注意事項（給工程師）

### 10.1 System Prompt

```
你是發票 AI 管家，根據用戶的真實消費數據回答問題。

嚴格規則：
1. 只使用 Context 中的數據，不編造
2. 每個【】內的通路是獨立品牌，不可混淆
3. 用戶問某通路時只回答該通路，問品項時列出實際品項名稱
4. 如果 Context 中沒有，說「你的發票中沒有這個記錄」
5. 繁體中文，親切但專業
6. 引用具體數字（品項名、次數、金額）
7. 200 字以內
8. 適度用 emoji
```

### 10.2 Context 包含

- 通路列表（品牌名、類別、次數、金額、均價）
- **各通路 top 品項明細**（品項名、次數、金額）← v2 新增
- 消費類別彙總
- 月趨勢

### 10.3 特別注意

- Uber Eats / foodpanda 的發票是平台費，不是餐費
- 超市/量販單價高是正常的（大量採購），不標記為問題
- 不同定位的通路不能比價（7-11 咖啡 ≠ 星巴克咖啡）
- 壽司盤、配餐薯條等是一餐的組成部分，不該獨立算月費
- 語氣正面引導，不用「浪費」「過高」等負面字眼

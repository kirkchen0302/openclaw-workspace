# 919 載具驗證碼失效 — 使用者旅程與產品優化

> 更新日期：2026-03-20

## 問題全貌

```
用戶在財政部網站/其他App 更改載具驗證碼
  ↓
App 內的驗證碼跟財政部不一致
  ↓
App 無法同步發票（919 錯誤）
  ↓
用戶開 App 看不到新發票
  ↓
覺得沒用 → 沉睡 → Policy 過期
```

**影響：120 萬人（21.3%），佔全部 Policy 過期的 37.2%**

## 產品優化 User Story

### 🔴 P0：止血（防止新增 919）

**US-1：919 偵測 + 即時推播**
> 當系統偵測到用戶的載具驗證碼與財政部不一致（919），立即推播通知：「您的載具驗證碼已更新，請在 App 內同步，1 分鐘就能恢復發票同步」，並提供一鍵更新入口。

**US-2：App 內領獎/匯款帳戶設定**
> 讓用戶不需離開 App 就能完成中獎領獎流程（設定匯款帳戶、確認領獎），減少用戶被導去財政部的機會。

**US-3：驗證碼到期預警**
> 偵測驗證碼即將到期，提前 30 天推播提醒用戶在 App 內更新，避免到財政部更新後忘記同步。

### 🟡 P1：減少外流

**US-4：驗證碼重設流程優化**
> 簡化「忘記驗證碼」的流程，讓用戶在 App 內 3 步完成重設，不需跳到財政部。

**US-5：載具歸戶功能**
> 在 App 內支援悠遊卡/icash/信用卡等載具歸戶到手機條碼。

**US-6：捐贈發票功能**
> 在 App 內支援捐贈發票給公益團體。

### 🟢 P2：長期

**US-7：多 App 衝突偵測**
> 偵測到用戶可能同時使用其他發票 App（919 頻率分析），推播提醒「請以發票存摺的驗證碼為準」。

**US-8：報稅季防護**
> 每年 5 月報稅季前，主動推播提醒用戶「報稅查發票請用 App，避免在財政部網站修改驗證碼」。

## 使用者旅程對比

**現在（壞路徑）：**
```
中獎 → 想領獎 → App沒這功能 → 搜尋到財政部 → 改了驗證碼 
→ 沒回App同步 → 919 → 沒發票 → 沉睡 → Policy過期
```

**優化後（好路徑）：**
```
中獎 → App推播領獎 → App內設匯款帳戶 → 不需離開App 
→ 驗證碼不會被改 → 發票持續同步 → MAVS
```

**919 發生後（修復路徑）：**
```
919偵測 → 立即推播「驗證碼不一致」→ 用戶點推播 
→ App內一鍵更新驗證碼 → 修復完成 → 發票恢復同步
```

## Android Source Code 分析（jeremylu-mb/invoice-android-readonly）

### 919 偵測機制

**偵測是被動的** — 只有用戶開 App 進入特定頁面時才檢查 `carrier.isValid`：

```kotlin
// BaseInvoiceBookViewModel.kt（發票簿）
InvoiceShared.carrier?.isValid == false → 顯示 919 對話框

// BaseInvoiceListViewModel.kt（發票列表）
it.carrier?.isValid == false → 顯示 919 對話框

// MemberProfileImpl.kt（會員首頁）
if (it.carrier?.isValid == false) openUpdateCarrierPasswordDialog.value = true
```

**沒有主動推播機制。** 工程師在 `MemberCarrierViewModel.kt` 的註解直接寫了：
```
// server 同步從 200 變 919，用戶收到推播後，點進來可能看不到 919 的狀態？
```

### 919 對話框

```
標題：發票同步失敗
內容：您過去曾在財政部變更過載具驗證碼，或使用過「忘記密碼」功能，
     因此目前無法在發票存摺繼續同步發票。請立即更新驗證碼，以恢復同步功能。
按鈕：前往更新驗證碼
```

- `cancelable = false`（不可取消，強制顯示）
- 來源有兩個：INVOICE（發票列表）和 CARRIER（載具管理）

### 更新驗證碼流程

```
用戶點「前往更新驗證碼」
  → LoginContainerActivity (UPDATE_CARRIER_PASSWORD)
  → UpdateCarrierPasswordScreen
  → 輸入手機號碼 + 新密碼 + 同意條款
  → memberRepository.updateCarrierPassword(password)
  → 成功：顯示「載具驗證碼更新成功，系統正在同步，最多需要一至兩天」
  → 失敗(INVALID_CREDENTIALS)：3 次後顯示「忘記密碼」對話框 → 導向財政部
```

### 從 code 發現的 4 個問題

**🔴 問題 1：只有被動偵測，沒有主動推播 919**
- 用戶不開 App 就永遠不知道 919
- 而 919 的用戶 87.9% 都沉睡（不開 App）
- 形成死循環：919 → 沒發票 → 不開 App → 不知道 919

**🔴 問題 2：更新成功後需要 1-2 天同步**
```xml
<string name="carrier_password_update_success_desc">
  系統正在同步您的發票與載具資料，最多需要一至兩天時間，請耐心稍候。
</string>
```
- 用戶更新完密碼後看不到即時效果
- 可能以為沒修好而放棄

**🟡 問題 3：密碼錯 3 次就導向財政部（循環）**
```kotlin
companion object {
    private const val MAX_INVALID_ATTEMPTS = 3
}
// 3 次後 → 顯示「忘記密碼」→ 導向財政部重設
// 但去財政部重設又會觸發新的 919！
```
- 用戶不記得在財政部改的密碼 → 輸錯 3 次 → 被導去財政部 → 在財政部重設 → 又觸發 919

**🟡 問題 4：對話框不可取消但體驗突兀**
- `cancelable = false` 確保用戶看到
- 但用戶可能只是想看發票，被強制擋住
- 可能導致用戶直接關 App

### 優化建議（基於 code 分析）

| # | 問題 | 建議 | 改動範圍 |
|---|---|---|---|
| 1 | 被動偵測 | 加入 FCM 推播：server 偵測到 919 立即推播 | Backend + FCM |
| 2 | 同步延遲 | 更新成功後立即觸發一次同步，並顯示進度 | `performActualUpdate()` |
| 3 | 密碼循環 | 3 次錯誤後提供「用手機號碼重設」而非導去財政部 | `handleInvalidCredentials()` |
| 4 | 對話框體驗 | 改成 Snackbar + 頂部提醒，不完全擋住內容 | `UpdateCarrierPasswordDialogCompose` |

## 根因確認（Code + 數據雙重驗證）

### 根因：App 自己把用戶導去財政部

```kotlin
// AddBankWebViewModel.kt — 領獎帳戶設定
fun getAddBankAccountSetting() {
    frontRepository.getAddBankAccountSetting()  // 從 server 拿 URL
    → Event.LoadWebView(url, javascript)        // WebView 打開財政部！
}
```

**App 內的「領獎帳戶設定」功能是用 WebView 載入財政部網站。** 用戶在財政部網站上被要求輸入/修改驗證碼，導致跟 App 不同步 → 919。

### 數據佐證

**1. 對獎日是 919 爆發觸發點**

| 日期 | 919 新增量 | vs 平時 |
|---|---|---|
| 25 號（對獎日） | 44,000 | 2.2x |
| 26 號 | **62,902** | **3.1x**（最高峰）|
| 27 號 | 52,354 | 2.6x |
| 平時 | ~20,000 | 基線 |

**2. 修好後 1-6 個月又壞（對應對獎週期）**

| 間隔 | 比例 | 對應 |
|---|---|---|
| 31-90 天 | 28.5% | 跨 1-3 次對獎日 |
| 91-180 天 | 33.0% | 跨 2-3 次對獎日 |
| 合計 | **61.5%** | 每次對獎都可能再壞 |

**3. 85% 修好後又壞**

| 狀態 | 人數 | 比例 |
|---|---|---|
| 修好沒再壞 | 108,848 | 15% |
| 修好又壞 | 625,987 | **85%** |

### 完整因果鏈

```
對獎日（25號）
  ↓ 用戶在 App 內對獎 → 中獎
  ↓ 點「領獎帳戶設定」
  ↓ App 用 WebView 打開財政部（AddBankWebActivity）
  ↓ 財政部要求輸入/修改驗證碼
  ↓ 用戶在財政部改了密碼
  ↓
App 內密碼跟財政部不一致 → 919
  ↓ 用戶下次開 App → 看到「發票同步失敗」對話框
  ↓ 點「更新驗證碼」→ 修好了
  ↓
下次對獎日（2個月後）→ 又領獎 → 又去財政部 → 又改密碼 → 又 919
  ↓ 85% 會再次循環 ♻️
```

### 根本解決方案

**不是修 919 的偵測或提醒，而是讓用戶不需要去財政部：**

| 方案 | 說明 | 影響 |
|---|---|---|
| **P0：領獎帳戶設定原生化** | 把 AddBankWebActivity 從 WebView/財政部 改成 App 原生功能，直接呼叫財政部 API | 從根本消除觸發 919 的場景 |
| **P0：WebView 攔截驗證碼修改** | 如果短期無法原生化，在 WebView 中偵測密碼修改行為，自動同步回 App | 減少 919 發生 |
| **P1：對獎日前推播提醒** | 每月 24 號推播「對獎請在 App 內操作，避免去財政部」 | 預防性 |

## 三、本週重點發現：919 根因分析完成

經 BQ 全量分析 + Android 原始碼交叉驗證：

**919 觸發來源拆解：**

| 來源 | 佔比 | 可控嗎 |
|---|---:|---|
| App 內 setBankAccount（領獎帳戶 WebView→財政部） | **18%** | ✅ 可修 |
| App 內忘記密碼→導去財政部 | 2.6% | ✅ 可修 |
| **App 外改密碼**（其他發票 App、瀏覽器、報稅） | **~55%** | ⚠️ 只能偵測 |
| 系統端批次偵測 | ~24% | ⚠️ 被動 |

**關鍵發現：**
1. **setBankAccount 是可修的 bug** — code 裡 `AddBankWebViewModel` 沒有 `handleResponse`（載具登入有但領獎帳戶沒有），WebView 打開財政部但不同步密碼回來
2. **85% 修好後又壞** — 每 2 個月對獎日就循環一次
3. **89% 的 919 用戶沉睡** — App 只有被動偵測（開 App 才知道），形成死循環
4. **大多數 919（~76%）來自 App 外** — 即使修了 setBankAccount，919 不會消失

**對獎日衝擊：** 開獎月 25-27 號 919 爆量是非開獎月的 **4-5 倍**

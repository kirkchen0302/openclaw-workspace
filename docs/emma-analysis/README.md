# 發票 App 會員斷點數據分析

## 專案概述
分析發票存摺 App 的會員生命週期，找出使用者斷點和流失原因。

## 資料來源
- **GCP 專案**：`production-379804`（會員/載具/Policy/發票/Session）、`invoice-bfd85`（Firebase/GA4）
- **BigQuery 表**：intermediate 層原始資料，非預計算 view
- **GA4 API**：property 158616188（驗證用）

## 核心發現

### 會員概況（截至 2026-03-17）
- 累計註冊：5,535,460
- DEACTIVATED（刪除）：28,992
- 未刪除會員：5,506,468

### 漏斗（既有會員，排除刪除/當月註冊）

```
既有會員      5,495,058 (100%)
  ↓ -6.3%  無載具
載具有效      5,150,430 (93.7%)
  ↓ -39.5% Policy過期        ← 最大斷點
Policy有效    2,979,147 (54.2%)
  ↓ -7.1%  近30天沒發票
30天有發票    2,588,575 (47.1%)
  ↓ -0.6%
三者皆有      2,544,685 (46.3%)
```

加入活躍條件後：
```
30天活躍      2,349,453 (42.8%)
MAVS          2,281,139 (41.5%)
```

### 關鍵洞察

1. **Policy 是最大斷點**：54.2% 有效，持續惡化（2025-04: 61.1% → 2026-03: 54.2%）
2. **因果關係**：先不活躍 → 才 Policy 過期（92.3%），不是反過來
3. **發票是留存關鍵**：有發票沉睡率 24.4%，沒發票 96.5%
4. **卸載率 54.1%**
5. **新註冊 MAVS 轉換率**：從 2025 的 ~80% 提升到 2026 的 ~99%

### 沉睡主因排序
1. 卸載 App（54%）
2. Policy 過期（45.8%）
3. 沒有發票（253 萬人）
4. 無載具（6.3%，不是主因）

### 最有救的群體
- **69 萬人**：已沉睡但 Policy 還有效，需立即喚回
- **8.7 萬人**：Policy 過期後還開 App 但沒續約，可能是 UX 問題

## 檔案說明
- `schema.md` — 欄位定義
- `api_field_mapping.md` — API ↔ BigQuery 欄位對照
- `daily_metrics_6m.sql` — 每日指標 SQL（6個月版）
- `daily_metrics_6m.csv` — 查詢結果
- `analysis_report.md` — 完整分析報告

## 欄位定義
- **MAVS**：近30日活躍 ∩ 載具有效(is_active=true) ∩ Policy有效(180天內)
- **HYVS**：載具有效 ∩ Policy有效 ∩ 近180天有載具發票
- **刪除**：status = DEACTIVATED
- **活躍**：近30天有 session_start（production session 表）
